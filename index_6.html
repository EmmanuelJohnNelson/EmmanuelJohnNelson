Teesside Regional Economic Data Dashboard

An interactive Power BI dashboard built to analyse and communicate Teesside's regional economic health — translating complex public datasets into actionable intelligence for policymakers and academic reviewers. Awarded distinction by faculty and recognised by industry partners at Tees Valley Combined Authority.

DistinctionAcademic Grade

Project Story — Step by Step
01
The Problem — A Region Without a Clear Mirror
Teesside's economic data was scattered across ONS datasets, local council reports, and industrial surveys — none of it in one place. Policymakers couldn't easily see how employment, GDP contribution, and investment flows related to each other over time. The goal was a single interactive dashboard that told that story clearly.
Multiple disconnected government data sources — ONS, NOMIS, and TVCA reports
02
Data Acquisition & Cleaning with Python
Three open government datasets were downloaded — ONS regional GDP figures, NOMIS employment statistics, and TVCA investment reports. Python (Pandas) was used to clean inconsistencies, standardise column names, resolve date format mismatches, and remove duplicates before loading into a structured SQL schema.
Python / Pandas pipeline — standardising and merging three regional datasets
import pandas as pd

gdp_df    = pd.read_csv('ons_teesside_gdp.csv')
employ_df = pd.read_csv('nomis_employment.csv')
invest_df = pd.read_csv('tvca_investment.csv')

for df in [gdp_df, employ_df, invest_df]:
    df['date']   = pd.to_datetime(df['date'], dayfirst=True)
    df['region'] = df['region'].str.upper().str.strip()

merged = gdp_df.merge(employ_df, on=['date','region'])
merged = merged.merge(invest_df, on=['date','region'])
print(f"Clean rows: {len(merged)} | Nulls: {merged.isnull().sum().sum()}")
03
PostgreSQL Schema Design & SQL Views
The cleaned data was loaded into a normalised PostgreSQL schema with three tables linked by a region-date composite key. SQL views were created for each indicator group, making Power BI ingestion fast and the underlying data reusable for future analysts.
PostgreSQL normalised schema — three tables, composite keys, reusable SQL views
04
Building the Power BI Dashboard
Power BI was connected directly to the PostgreSQL views. Six report pages were designed: Regional Overview, GDP Trends, Employment Heatmap, Investment Flow, Sector Comparison, and a Year-on-Year Summary. DAX measures were written for running totals, YoY percentage change, and cross-sector correlations. Slicers allowed filtering by year, sector, and sub-region.
Power BI dashboard — six report pages with DAX measures, slicers, and KPI cards
05
Faculty Presentation & Industry Recognition
The completed dashboard was presented to a panel of faculty assessors and industry reviewers from Tees Valley Combined Authority. The panel praised the clarity of the data storytelling, the DAX logic quality, and the practical applicability for real regional policy decisions. The project was awarded a distinction — the highest grade available.
Presentation to faculty and Tees Valley Combined Authority reviewers — awarded Distinction

E-Commerce Sales Forecasting Model

A production-grade time-series forecasting system built to predict 6-month forward sales for an e-commerce operation — combining classical ARIMA with machine learning to achieve 90% forecast accuracy and eliminate stock-planning guesswork.

90%Forecast Accuracy
6 moPrediction Horizon

Project Story — Step by Step
01
The Challenge — Predicting What's Next
An e-commerce dataset spanning 18 months of daily sales records was provided for academic assessment. The challenge: build a model that could reliably forecast the next 6 months, providing enough lead time for procurement and inventory planning. Target accuracy was 85%; the stretch goal was 90%.
The client challenge: accurate 6-month sales forecasting to streamline inventory management
02
Exploratory Data Analysis & Stationarity Testing
The raw sales series showed clear weekly seasonality and a long-term upward trend — both needed handling before modelling. The Augmented Dickey-Fuller test confirmed non-stationarity, so first-order differencing was applied. ACF and PACF plots were then analysed to determine correct ARIMA(p,d,q) parameters.
EDA phase — ACF/PACF plots, seasonality decomposition, and stationarity testing
from statsmodels.tsa.stattools import adfuller

result = adfuller(sales_series)
print(f"ADF Statistic: {result[0]:.4f}")
print(f"p-value: {result[1]:.4f}")  # p > 0.05 → non-stationary

# Apply first-order differencing → re-test
sales_diff = sales_series.diff().dropna()
# p-value now < 0.01 → stationary ✓
03
Model Comparison — ARIMA vs SARIMA vs XGBoost
Three model candidates were evaluated on a 20% hold-out test set using RMSE and MAPE. SARIMA(2,1,2)(1,1,1,7) outperformed ARIMA and XGBoost by capturing both trend and weekly seasonality, delivering a MAPE of 9.8% — translating to 90.2% accuracy.
Three-model cross-validation — SARIMA emerged as the clear winner at 90.2% accuracy
04
Building the 180-Day Forecast with Confidence Intervals
The final SARIMA model was fitted on the full 18-month training set and used to generate a 180-day forward forecast with 95% confidence intervals. The widening confidence band gave stakeholders an honest view of forecast uncertainty rather than false precision.
180-day SARIMA forecast with 95% confidence band — actuals vs predicted
from statsmodels.tsa.statespace.sarimax import SARIMAX

model  = SARIMAX(sales_train, order=(2,1,2), seasonal_order=(1,1,1,7))
fitted = model.fit(disp=False)

forecast    = fitted.get_forecast(steps=180)
forecast_df = forecast.summary_frame(alpha=0.05)
print(forecast_df[['mean','mean_ci_lower','mean_ci_upper']].head())
05
Tableau Visualisation for Stakeholders
The forecast output was exported to CSV and visualised in Tableau as a dual-axis chart — historical actuals alongside the forward projection with its confidence band. Designed for non-technical stakeholders making stock and budgeting decisions, this visual was the centrepiece of the academic presentation and received particular praise from assessors.
Tableau dual-axis forecast chart presented to stakeholders and academic assessors

Student Retention Analytics System

A full-stack web application that uses machine learning to score individual students by dropout risk — giving academic advisors an early-warning dashboard to intervene before students disengage. Awarded top project distinction by the Teesside University faculty panel.

87%ML Accuracy
TopDistinction Grade

Real-timeRisk Scoring
Project Story — Step by Step
01
The Problem — Students Slipping Through the Cracks
Universities lose a significant portion of students to dropout each year — and most interventions happen too late, after a student has already disengaged. The brief was to build a system that surfaces at-risk students early using academic and engagement data, so advisors can act while there's still time to help.
The problem: students disengaging silently before advisors can intervene
02
Database Design — Modelling a Student's Academic Life
PostgreSQL was used to design a normalised 5-table schema: students, modules, grades, attendance, and engagement_events. Foreign key relationships were carefully mapped so any student's full academic history could be retrieved in a single JOIN query — the foundation the ML feature pipeline depended on.
5-table PostgreSQL schema — students, grades, attendance, modules, engagement_events
-- Core schema (simplified)
CREATE TABLE students (
    id         SERIAL PRIMARY KEY,
    name       VARCHAR(100),
    programme  VARCHAR(80),
    year       INT
);
CREATE TABLE engagement_events (
    student_id  INT REFERENCES students(id),
    event_type  VARCHAR(50),
    event_ts    TIMESTAMPTZ,
    platform    VARCHAR(30)
);
03
Feature Engineering & ML Pipeline
Twelve features were engineered: rolling average grade, 4-week attendance rate, days since last login, submission rate, and missed deadlines count. A Random Forest classifier was selected after cross-validation. SMOTE was applied to address class imbalance between at-risk and non-at-risk students, achieving 87.4% accuracy.
Random Forest + SMOTE pipeline — 12 engineered features, 87.4% test accuracy
from sklearn.ensemble import RandomForestClassifier
from sklearn.pipeline import Pipeline
from imblearn.over_sampling import SMOTE

pipeline = Pipeline([
    ('smote', SMOTE(random_state=42)),
    ('model', RandomForestClassifier(n_estimators=200, max_depth=8, random_state=42))
])
pipeline.fit(X_train, y_train)
print(f"Accuracy: {pipeline.score(X_test, y_test):.2%}")
# → Accuracy: 87.40%
04
Flask REST API — Serving Predictions
The trained model was serialised with joblib and wrapped in a Flask REST API. Two endpoints were built: GET /students to retrieve the student list, and POST /predict to return a real-time dropout risk score (0–100). JWT authentication restricted access to authorised staff only.
Flask REST API with JWT auth — two endpoints serving real-time risk scores
from flask import Flask, request, jsonify
import joblib

app   = Flask(__name__)
model = joblib.load('retention_model.pkl')

@app.route('/predict', methods=['POST'])
def predict():
    features   = request.json['features']
    prob       = model.predict_proba([features])[0][1]
    risk_score = round(prob * 100, 1)
    level = 'HIGH' if prob > 0.6 else 'MEDIUM' if prob > 0.35 else 'LOW'
    return jsonify({'risk_score': risk_score, 'risk_level': level})
05
React Dashboard & Top Distinction Award
The React.js frontend displayed a sortable table of students colour-coded by risk level — red (high), amber (medium), green (low) — with drill-down cards showing the key factors behind each score. The faculty panel praised the explainability layer particularly: advisors could see exactly why a student was flagged, not just a number. The project was awarded the top distinction of the entire cohort.
React.js dashboard — colour-coded risk table with explainability drill-down cards
